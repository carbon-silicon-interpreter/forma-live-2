# FORMA LIVE Demo 结构与产品分析

> 范围说明：本文只基于 `forma-live` 目录中的源码、配置文件、构建产物和同目录文档整理，不引用仓库其他目录，也不参考外部资料。

## 1. 结论先行

从这个 demo 的代码和页面结构看，`forma-live` 不是一个单纯的文档编辑器，也不只是一个静态站点模板。

它更像是一个：

**由知识库驱动、由 AI 参与生成、面向客户交付与运营的 live site / living handbook 平台 demo。**

如果用更通俗的话说，它展示的是这样一条链路：

1. 把 PDF、Word、TXT、PPT 等资料放进知识库。
2. 由 AI 推荐应该生成哪些内容模块。
3. 在模块级工作台里继续编辑最终内容、中间物料和“思考过程”。
4. 发布成一个可阅读、可问答、可下载、可追踪运营数据的客户侧站点。

所以它最接近的产品形态是：

**AI 驱动的知识交付空间 / 活态手册 / 可运营内容站。**

不是传统意义上的：

- 纯文档工具
- 纯 CMS
- 纯官网建站器
- 纯问卷工具

## 2. 目录和代码结构

### 2.1 顶层目录信号

这个目录里同时存在：

- `src/` 源码
- `dist/` 构建产物
- `node_modules/` 依赖
- `package.json`、`vite.config.js`、`vercel.json`
- 两份产品向文档：
  - `FORMA_LIVE_现有框架理解.md`
  - `LIVING_HANDBOOK_CONCEPT.md`

这说明它不是只做了一份设计稿，而是一个已经能本地运行、也考虑过部署的 demo 快照。

### 2.2 技术栈

从 `package.json` 看，技术栈很轻：

- React 19
- React Router DOM 6
- Vite 7
- Tailwind CSS 4
- lucide-react 图标库

从 `vercel.json` 看，部署方式是单页应用路由重写，说明它本来就被设计成一个前端单页 demo，而不是多页后端系统。

### 2.3 源码结构

源码的主结构比较清楚：

- `src/App.jsx`
  - 路由入口
  - 全局状态入口
- `src/config/mockData.js`
  - 所有 demo 数据
  - 包括知识库素材和页面模块
- `src/views/DashboardView/`
  - 工作台首页
- `src/views/EditorView/`
  - 总编辑页
  - 模块编辑页
- `src/views/PreviewView/`
  - 最终站点预览页
  - 运营视图
- `src/components/blocks/`
  - 问卷、PPT、富文本渲染器
- `src/components/common/`
  - 背景氛围层

## 3. 运行结构

### 3.1 路由结构

`src/App.jsx` 里只有四个核心路由：

- `/`
  - Dashboard
- `/editor`
  - 总编辑页
- `/editor/:id`
  - 单模块编辑页
- `/site/:id`
  - 最终预览页

这个结构本身已经说明产品主流程是：

**项目入口 -> 结构编排 -> 模块细编 -> 最终交付 / 运营。**

### 3.2 顶层状态

顶层状态只有四类：

- `materials`
  - 知识库素材
- `pages`
  - 最终站点里的模块集合
- `activeDoc`
  - 预览页当前聚焦模块
- `previewMaterial`
  - 编辑器里当前要预览的源资料

这意味着这个 demo 的核心数据模型并不复杂，主要围绕两个对象转：

- 素材
- 模块

最终站点只是把这些模块串起来渲染。

## 4. 产品主流程

### 4.1 Dashboard：项目/空间入口

首页顶部写的是 `Living Handbook`，但卡片文案写的是：

- 创建营销/交付空间
- 中小企业用工合规解决方案
- 活态营销空间
- 支持多客户复用

这说明首页表达的不是“写一本手册”，而是：

**创建一个面向某个业务场景的交付空间。**

首页的动作也验证了这一点：

- 去编辑
- 去运营

所以 Dashboard 更像“项目工作台”或“交付空间列表”，不是文档库。

### 4.2 Editor：知识库到模块结构的编排台

编辑页由三部分组成：

- 左侧：知识库
- 中间：模块列表
- 底部：AI 输入框

它还带了三类关键动作：

- AI 推荐架构
- 手动添加模块
- 部署 / 预览 / 运营

这里最重要的不是页面排版，而是：

1. 从知识库中挑素材。
2. 让 AI 推荐该生成哪些模块。
3. 给每个模块绑定关联资料。
4. 批量生成模块骨架。

所以 Editor 的真正职责是：

**把知识素材编排成客户可见的内容模块结构。**

### 4.3 Module Editor：模块级 AI 生产工作台

单模块编辑页不是普通文本编辑器，而是三栏逻辑：

- 最终产物预览
- 中间层物料
- 思考过程

右边还有一个固定 `AI Copilot` 对话面板。

这说明它要表达的不是“改文案”，而是：

**AI 参与模块生成全流程，并且这个流程是可解释、可复查、可重写的。**

不同模块类型会有不同的编辑呈现：

- 视频模块
  - 分镜脚本
  - 视觉素材
- PPT 模块
  - 幻灯片成品
  - 结构化大纲
- 问卷模块
  - 手机端问卷预览
  - 题目权重表
- 图文页模块
  - 成稿预览
  - 知识片段池

这个页面已经很明确地把产品定位往“AI 内容生产台”推，而不是“手册章节编辑器”。

### 4.4 Preview / Operate：面向客户的 live site 和运营台

预览页不是单纯前台页面，它其实分成两层模式：

- `preview`
  - 前端浏览
- `operate`
  - 运营后台

前台侧有：

- 项目概览
- 模块化内容展示
- 资料库
- 右侧常驻 AI 顾问

运营侧有四个 tab：

- 前端预览
- 深度洞察
- 详细数据
- 对话记录

这说明最终产物不是一个“看完就结束”的静态页，而是：

**一个会被持续监控、优化、转化运营的活态交付站点。**

## 5. 核心对象模型

### 5.1 Materials：知识源

`mockData.js` 中的素材对象有这些属性：

- `id`
- `name`
- `type`
- `visible`
- `tags`
- `blocks`

当前素材类型能看出来至少覆盖：

- pdf
- doc
- txt
- ppt

编辑器里“添加素材”的入口也进一步暗示了输入源范围：

- 本地文件
- 网络链接
- 手动文本
- 第三方知识库

所以 `materials` 的产品含义很清楚：

**知识库 / 素材池 / 生成输入源。**

### 5.2 Pages：交付模块

`pages` 是最终展示给用户的模块数组。

当前 demo 里已经落了几种模块类型：

- `questionnaire`
- `video`
- `ppt`
- `page`

编辑器里定义的“标准模块列表”还包括：

- FAQ 页
- 图文案例页
- 讲解视频页
- 问卷调查页
- PPT 介绍页

这说明产品的基础交付单元不是“文章章节”，而是：

**多模态模块。**

### 5.3 Site：运行态交付对象

最终预览页会把：

- 项目概览
- 多个模块
- 资源下载区
- AI 顾问

全部编排到一个连续站点里。

所以 `site` 不是简单的页面集合，而是：

**可阅读、可对话、可下载、可追踪的运行态交付空间。**

## 6. 这个 demo 到底在卖什么

如果只根据当前代码和 mock data，不看外部背景，这个 demo 当前表达的业务价值是：

**把服务行业的知识素材，生成一个面向客户的可交付、可讲解、可转化、可运营的活态站点。**

当前 demo 里的内容形态包括：

- 企业自测问卷
- 长视频讲解
- PPT 培训课件
- 标准合同库
- 下载型资料库
- AI 顾问问答
- 运营洞察和数据表
- 会话审查

这比“做一本 handbook”更接近一个：

**服务销售与交付一体化的内容产品。**

## 7. 从角色上看，这是一个双端产品

### 7.1 内部角色

内部角色是内容运营者 / 销售 / 交付人员，他们做的事包括：

- 导入资料
- 选模块
- 让 AI 生成结构
- 逐模块编辑
- 分享链接
- 生成传播卡片
- 运营复盘

### 7.2 外部角色

外部角色是客户或学员，他们看到的是：

- 一个连续浏览的内容站点
- 一个 AI 顾问
- 一个可填写的问卷
- 一组可预览/下载的资料

因此从产品关系上看，它接近：

**B 端创建，C 端消费，B 端持续运营。**

## 8. 这个 demo 最明显的产品信号

### 8.1 它非常强调 AI 不是装饰

AI 在这个 demo 里承担了至少四层角色：

- 推荐模块结构
- 生成模块骨架
- 在模块内协同修改
- 在前台承担问答和导航

运营页里的洞察和建议，也继续把 AI 放在优化角色上。

换句话说，AI 在这里不是“聊天按钮”，而是产品主逻辑的一部分。

### 8.2 它强调“从知识到交付”的过程

所有页面都在反复表达同一件事：

- 知识库
- 模块化生成
- 最终 live site
- 运营数据回流

这是一条很完整的产品叙事链。

### 8.3 它强调交付后的运营

这点很关键。

很多内容产品 demo 到“发布可看”就结束了，但这个 demo 明确往后走了一步：

- 洞察
- 数据
- 留资
- 对话审查
- 内容优化建议

因此它卖的不只是“生成内容”，而是：

**生成后还能持续优化和转化。**

## 9. 当前 demo 里存在的两层混用

这是理解这个 demo 时最需要注意的一点。

### 9.1 命名混用

代码里同时出现了这些定位词：

- Living Handbook
- Living Site
- 营销/交付空间
- Forma 知识交付平台

这意味着当前 demo 的产品概念还没有完全收口。

但从页面结构看，最稳定的骨架其实是：

**交付空间 / live site。**

`Living Handbook` 更像是上层概念包装，而不是底层系统结构名。

### 9.2 业务场景混用

内容侧也混着两套业务叙事：

一套偏法务 / 用工合规：

- 法保网品牌介绍
- 用工合规白皮书
- 劳动关系 FAQ
- 竞业限制协议
- 法务 AI 顾问

另一套偏财税 / 代理记账 / 审计：

- 企业财税健康自测问卷
- 财税审计资料库
- 福马财税介绍
- 财税风险排查

所以更准确的判断不是“它就是法务产品”或“它就是财税产品”，而是：

**它在用服务行业知识交付场景做 demo，但当前样例内容还处于法务与财税混编状态。**

## 10. 这不是完整产品，而是高保真前端 demo

从实现上能明显看出很多能力仍然是模拟态：

- 数据全部来自 `mockData.js`
- AI 推荐相关度是哈希伪算法
- 上传素材是 `setTimeout` 模拟
- AI 对话和内容更新是 `setTimeout` 模拟
- 运营数据是硬编码或简单公式生成
- 分享链接是固定字符串

另外，整个项目没有看到：

- 后端接口层
- 鉴权体系
- 持久化存储
- 真正的文档解析服务
- 真正的模型调用

所以这个目录表达的是：

**产品结构、交互逻辑和视觉语言已经相当完整，但系统能力仍然主要停留在前端演示层。**

## 11. 一句话定义这是什么产品

如果需要给这个 demo 一个尽量贴近目录内容的定义，我会这样写：

**Forma Live 是一个面向服务销售与客户交付场景的 AI 知识交付平台：它把知识库素材编排为多模态内容模块，生成可阅读、可问答、可下载、可运营的 live site / living handbook。**

如果再压缩一点，就是：

**一个 AI 驱动的活态交付站点生成与运营 demo。**

## 12. 建议如何理解“Living Handbook”

只根据这个目录里的内容，我更倾向于这样理解：

- `Living Handbook`
  - 是用户更容易理解的表达层
- `Live Site / 交付空间`
  - 是更真实的产品骨架

也就是说：

**这个 demo 本质上不是“电子手册工具”，而是“可被包装成 living handbook 的知识交付空间”。**

## 13. 可作为后续沟通的简版口径

如果后面需要一句对外解释，可以直接用下面这版：

> `forma-live` 这个 demo 展示的是一个 AI 驱动的知识交付平台。内部人员把资料放进知识库后，系统会推荐并生成问卷、视频、PPT、图文页等模块，最后发布成一个带 AI 顾问、资源下载和运营分析能力的客户侧 live site。它可以被理解成一本“活的 handbook”，但底层更像可运营的交付空间，而不是传统文档工具。
