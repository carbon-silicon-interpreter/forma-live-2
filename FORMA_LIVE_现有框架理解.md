# Forma Live 现有框架理解

*仅基于 `forma-live` 目录内现有 demo 代码与同目录概念文档整理。*
*不引用仓库其他文档，不外推长期路线。*

---

## 1. 结论

这个 demo 当前表达的**产品本体**不是“做一本 handbook 的工具”，而是：

> **一个由知识库驱动的、可生成多模块交付页面、可被 AI 协助、可被运营的数据化 live site / 交付空间。**

更具体一点：

- 它先有一组知识素材
- 再由 AI 帮你组织成不同模块
- 每个模块可以继续深度编辑
- 最后发布成一个可阅读、可问答、可运营的 live site

`living handbook` 在这个 demo 里应该扮演的是：

- **表达层**
- **阅读隐喻**
- **包装概念**

而不是把这套产品框架整个改写掉。

---

## 2. 现有 demo 的产品框架是什么

从路由和状态结构看，这个 demo 的主框架已经很清楚：

### 2.1 Dashboard

入口页承载的是“项目 / 空间”视角。

这里不是一个单纯文档列表，而是一个**交付空间列表**：

- 有项目卡片
- 有更新时间
- 有“配置 / 运营”入口
- 指向的是后面的编辑与运营流程

从文案上看，虽然顶部已经写了 `Living Handbook`，但卡片文案依然是：

- “创建营销/交付空间”
- “活态营销空间”
- “支持多客户复用”

这说明当前骨架的中心仍然是：

> **围绕某个场景创建一个可交付、可复用、可运营的空间。**

### 2.2 Editor

编辑页的结构非常稳定，也非常像这个 demo 的真实核心：

- 左侧：`知识库`
- 中间：`模块列表`
- 顶部：`预览 / 部署 / 运营`
- 底部：AI 输入框，负责推荐新增模块
- 弹窗：`AI 推荐架构 / 手动配置`

这说明当前产品不是一个“写手册”的编辑器，而是一个：

> **把知识库编排成模块化交付空间的工作台。**

这里的重点不是页面排版本身，而是：

- 根据知识素材推荐模块类型
- 给模块绑定引用资料
- 允许 AI 自动生成页面框架
- 允许人工手动配置

也就是说，Editor 的真正职责是：

> **做“知识 -> 模块 -> 交付结构”的编排。**

### 2.3 Module Editor

单模块编辑页更进一步说明了这个产品不是静态文档工具。

它不是只有一个“编辑文本”的视图，而是明确拆成了三个层：

- `最终产物预览`
- `中间层物料`
- `思考过程`

再加上右侧的 `AI Copilot`，它表达的是一种非常明确的工作方式：

> **AI 不是只帮你润色最终页面，而是在参与整个生成过程。**

这一页真正表达的不是“手册章节编辑”，而是：

- 一个模块有自己的生成逻辑
- 有自己的中间物料
- 有可解释的链路
- 有可对话的修改过程
- 最终可测试、可发布

因此它本质上是：

> **模块级的 AI 生成工作台。**

### 2.4 Preview / Operate

预览页是最终对外的 live site。

但它也不是简单的“前台页面”，因为它同时内置了两种模式：

- 阅读/预览模式
- 运营模式

并且页面内部已经具备这些结构：

- `项目概览`
- 多个内容模块
- `资料库`
- 右侧常驻 AI 顾问
- `深度洞察 / 详细数据 / 对话记录`

所以这个阶段的产品完成闭环是：

> **知识库 -> 模块 -> live site -> 用户阅读/问答 -> 运营洞察**

这就是当前 demo 的真正产品框架。

## 3. 这个 demo 当前在卖什么

如果只看现有 mock data 和页面逻辑，这个 demo 当前卖的是一个很具体的业务场景：

> **把法务/财税服务的知识素材，组织成一个面向客户或学员的活态交付空间。**

这个空间包含：

- 品牌介绍
- 白皮书知识点
- FAQ
- 合规协议
- 诊断问卷
- 讲解视频
- PPT 课件
- 标准合同库
- AI 顾问
- 运营数据

所以它更像：

- 销售交付页
- 培训交付页
- 客户教育页
- 服务资料站

这些都比“手册本身”更接近当前 demo 实际在表达的东西。

---

## 4. 核心对象是什么

只基于这套 demo，我认为当前产品里有 3 个核心对象：

### 4.1 Materials

这是知识源。

特点：

- 有类型（pdf/doc/txt/ppt）
- 有标签
- 有可见/不可见
- 可以被多个模块引用
- 可以预览

它在产品里的角色是：

> **知识库 / 素材池 / 生成输入源**

### 4.2 Pages / Modules

这是交付结构单元。

当前模块类型包括：

- questionnaire
- video
- ppt
- page

这说明产品的基本单位不是“章节文本”，而是：

> **多模态模块**

这些模块可以被 AI 生成、编辑、测试、发布。

### 4.3 Site

这是最终交付给用户的 live site。

它不是单页静态页面，而是：

- 有模块导航
- 有 AI 助手
- 有 intro / resources 补位
- 有运营模式

所以 `site` 在这个 demo 里更像：

> **运行态交付对象**

---

## 5. AI 在现有框架里的角色

这个目录里的 AI 不是装饰性的。

它至少承担了 4 层角色：

### 5.1 架构推荐

在 Editor 里，AI 根据已有资料推荐模块类型和关联素材。

### 5.2 模块生成

AI 能生成新模块框架，并把它加入模块列表。

### 5.3 模块深度协作

在 Module Editor 中，AI 参与：

- 中间物料生成
- 思考过程展示
- 对话式修改
- 产物同步

### 5.4 运行态导览

在 Preview 中，AI 会根据用户问题把他带到：

- 问卷
- 合同库
- 视频
- 资料库
- 动态案例

这说明当前 AI 的作用不是“陪聊”，而是：

> **协助生成 + 协助导航 + 协助转化**

这点是现有框架很重要的组成部分，不能因为加 handbook 概念就弱化掉。

---

## 6. living handbook 应该加在哪一层

我现在的理解是：

> **living handbook 应该加在“体验表达层”，而不是加在“产品骨架层”。**

也就是：

- 不改变产品主流程
- 不改变对象模型
- 不改变核心页面职责
- 只改变用户如何感知这套东西

### 6.1 可以增加 handbook 感的地方

可以加强的地方：

- Dashboard 的项目命名方式
- 模块在预览页里的章节感
- 导航的目录感
- 阅读路径的连续感
- 封面页 / 开篇页 / 附件页的手册化表达
- 模块之间的“上一章 / 下一章”
- 分享给别人时的“像一本数字手册”感受

### 6.2 不该改变的地方

不该改变的地方：

- `知识库 -> 模块列表 -> 模块编辑 -> 预览/运营` 这条主链路
- 模块化、多模态的结构
- 模块级 AI 生成逻辑
- 预览态和运营态的双模结构
- “活态交付空间”的底层逻辑

## 7. 这个目录里的概念文档，应该怎么理解

同目录里的 `LIVING_HANDBOOK_CONCEPT.md` 已经开始尝试把它改写成 handbook 叙事。

但从代码看，有一个很重要的事实：

> **代码里的产品框架没变，概念文档只是试图重新包装它。**

这意味着：

- 概念文档不是在要求我们推翻现有 demo
- 而是在要求我们把现有 demo 讲得更像 “living handbook”

所以正确的理解不是：

“把产品改造成 handbook 产品”

而应该是：

“保留现有 live site / 交付空间框架，让它在用户看来更像一本可交互、可运营、可被 agent 协助的 living handbook”

---

## 8. 后续改动的原则

如果后面继续改 `forma-live`，我认为要遵守下面几条：

### 8.1 产品框架不动

保留：

- Dashboard
- Editor
- Module Editor
- Preview / Operate

### 8.2 核心对象不动

保留：

- materials 作为知识源
- pages/modules 作为交付单元
- site 作为运行态结果

### 8.3 living handbook 只做增层

增加的是：

- 命名
- 目录感
- 章节组织感
- 封面/附件/插页感
- 阅读与分享感

而不是重写成另一套产品逻辑。

### 8.4 agent 和运营层不能弱化

因为当前 demo 最有价值的地方之一就是：

- 不是静态内容
- 不是一次性交付
- 不是纯展示页

一旦把它只做成“好看的数字手册”，就把现有框架的核心丢掉了。

---

## 9. 一句话对齐

如果只基于 `forma-live` 当前 demo，我会用这句话来对齐后续设计：

> **Forma Live 是一个由知识库驱动的活态交付空间；Living Handbook 只是让这套空间更容易被理解、阅读和分享的表达方式。**

再短一点：

> **框架是 live site / 交付空间，handbook 是概念包装，不是产品骨架。**
